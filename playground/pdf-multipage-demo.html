Making a multi-page PDF from HTML

This document shows how to set things up in order to get a proper multi-page PDF using Kendo UI. Click “Generate PDF”. The PDF intentionally looks a bit different from what you see in the browser, to show that you can customize PDF output independently.

See the documentation for the API info, and the source of this demo for a working example.

TL;DR

  1. Declare and use TrueType fonts.
  2. Make sure the element you pass to drawDOM has no border, padding or margin.
  3. Don't use too wide elements that cannot be resized.
  4. Use only px, em or percentage units in CSS. Declare a base font size on <body> in pixels, and use relative units for other elements.
  5. Use the kendo-pdf-document CSS selector to customize PDF output.
  6. Tall elements that have no text content cannot be split across pages.
  7. Lorem ipsum. ☺

Basic code

To enable automatic page breaking, you must pass the paperSize option to kendo.drawing.drawDOM. This demo uses the following code:

function makePDF() {
  kendo.drawing.drawDOM("#content", {
    paperSize: "A4",
    forcePageBreak: "H1:not(:first-child)",
    margin: {
      top: "3cm", left: "4cm", right: "4cm", bottom: "3cm"
    },
    template: $("#page-template").html()
  }).then(function(group){
    kendo.drawing.pdf.saveAs(group, "loremipsum.pdf");
  });
}

We pass to drawDOM the innermost element that contains the text we want to display in PDF. That element (#content) has no border, no padding and no margin. The page breaking algorithm will arrange things such that it will be displayed at the desired paper size and margins.

Fonts

You must declare and use TrueType fonts. Without this, the layout will be invariably broken. This demo uses the “DejaVu” font family, which is bundled with Kendo UI. Since this particular font is already declared in kendo.common.css, we only need to supply appropriate font-family rules so that it's used on our elements.

You may use any other fonts, as long as you provide the TrueType files on your server, and the CSS @font-face rules. Please see the documentation.

Remember that in order for Kendo to be able to dig your CSS files for these declarations, the CSS must be loaded from the same domain as the application (if that's not the case, you must manually declare the URLs to the TrueType files using kendo.pdf.defineFont). Fonts must also reside on the same domain, or else the server hosting them must provide the appropriate CORS headers — otherwise Kendo will not be able to load them.

Images

If your document contains any images, they must reside on the same domain as the application, or else the server hosting them must provide the appropriate CORS headers — otherwise Kendo will not be able to draw them (sounds familiar already?). In short, hosting things on another domain is certainly possible, but be prepared for some headaches. There is nothing we can do to simplify this.

CSS units

In general, if you target PDF output the only unit safe to use in CSS is px (pixels). Please see the documentation for details. However, you can safely use relative units too, such as percentage or em — but the values they relate to should ultimately be based on pixels for a proper layout.

Sizes

In general, what is displayed on screen will look too big in PDF for, say, an A4 paper. For example a 14px font size, which is a good choice for screen, will be 14pt (typographic point) in PDF, which is approximately 0.5cm — will look awfully big.

In order to scale down the text in PDF while displaying it at 14px on screen, this demo uses the following code:

/* in CSS: */
#main {
  font: 14px "DejaVu Serif";
}
/* next, for any other elements such as H1, H2 etc.,
   we use percentage values: */
h1 {
  font-size: 140%;
}

/* this rule applies only to the PDF output: */
kendo-pdf-document {
  font-size: 70%;
}

<!-- in HTML: -->
<div id="main">
  <div id="content">
    ... page content here...
  </div>
</div>

When we render the PDF, we draw the #content element, rather than #main. The page breaking algorithm will place a copy of the DOM content in a <kendo-pdf-document> tag, so the rule above will apply in the PDF output without affecting anything that the user sees in browser. Note that, since except the base 14px font size which is declared on the #main element, all other values are percentages, they will cascade properly in PDF. For example, ordinary text will be displayed at 14pt * 70%, and H1 text will be displayed at 14pt * 70% * 140%, producing the same aspect ratio as on screen.

Here is an image:

If we displayed in PDF the image in full size, it would be huge, and truncated on the right margin. In order to scale it down in PDF we use this CSS:

kendo-pdf-document .center-image img {
  max-width: 70%;
}

Thus, it will occupy at most 70% of the available page width.

Customizing the PDF output

You already saw it — using the kendo-pdf-document selector we can place CSS rules that apply only in PDF, and not on screen. For example this page uses:

kendo-pdf-document p {
  text-align: justify;
}

in order to justify paragraph text in PDF. There are some things to be aware of here:

  • You should not add images (via background-image) that are not already displayed on screen. That might or might not work, depending on whether those images are already cached and the alignment of the planets.

  • You should not use fonts that are not already rendered on screen. This is almost guaranteed to not work, because we will begin generating the PDF before the browser finishes the layout of our <kendo-pdf-document> tag. There is no DOM event to tell us “the layout is ready, proceed now”, so this cannot be fixed.

Page layout

You can notice that in this demo we didn't pay particular attention to layout. The page breaking algorithm will place the content in a properly sized element, so even if this text appears wide on screen, it's resized to fit on an A4 page with a 4cm horizontal margin in PDF.

Beware of very wide elements that are not resizable. For example if you have a table, or an image, and you manually declare a 1000px width, that will appear truncated in the PDF. Try to use relative units, like percentages. Also, make sure the content can actually fit on page width.

Keep in mind that page breaks can be inserted only in text nodes, or right before an element. An image will never be split — instead, if it falls on the page margin, it will be moved onto the next page. And if it doesn't fit on the page height, it will be truncated. Similar for a very tall element, say a div, which does not contain any text.

Page template

By using the template option we can specify a piece of HTML to be included in every page. This is how we display headers and footers on this page, as well as the ♥ Kendo ☺ watermark. Please see the documentation for more information about page templates.

The template is inserted in the DOM of each page, after the page contents, and it's an absolutely positioned element. The trouble with this is that it's rendered above the page contents. To make the watermark show below the contents we use the following CSS code:

kendo-pdf-document #content {
  position: relative;
  z-index: 10;
}
kendo-pdf-document .page-template {
  z-index: 0;
}

Normally the z-index lines should be enough, but we must also add position: relative to the page content; otherwise it would still show up below the watermark, due to some bizarre rules of how z-index works.

Forced page breaks

The next section starts on a new page, no matter where this one ends. This is because we included this option:

forcePageBreak: "H1:not(:first-child)"

and the next section is a <h1> tag (and it's not the first). By the way, the useful content of this document ends here, thank you for reading.
